# Домашнее задание к занятию «Микросервисы: подходы»

Вы работаете в крупной компании, которая строит систему на основе микросервисной архитектуры.
Вам как DevOps-специалисту необходимо выдвинуть предложение по организации инфраструктуры для разработки и эксплуатации.


## Задача 1: Обеспечить разработку
Выбор: Yandex Cloud + Gitlab CI/CD
| Критерий | Yandex Cloud |
|:---:|:---:|
| Облачная система | Yandex Cloud |
| Git VCS |  YC Container Registry + Git |
| Репозиторий на сервис | + | 
| Сборка по событию | push,merge, tag |
| Сборка по кнопке | + |
| Настройки к каждой сборке | Configuration parameters, env, secrets |
| Создания шаблонов для различных конфигураций сборок | Pipeline templates + YAML |
| Секреты | Lockbox (managed secrets) + IAM|
| Несколько конфигураций для сборки из одного репозитория |  Multiple pipelines + filters |
| Кастомные шаги | YC CLI + Cloud Functions + containers |
| Docker-образы | Использование собственных build-образов |
| Self-hosted runners | + |
| Parallel builds | Параллельные стадии + auto-scaling группы раннеров в K8s |
| Parallel tests | Распределение тестов по подам в YC Kubernetes + sharding |

Для реализации CI/CD я выбрал Yandex Cloud CI/CD, поскольку он обеспечивает нативную интеграцию с облаком и безопасное управление секретами через Lockbox. 
Для сбора логов в микросервисной архитектуре я предлагаю использовать Loki + Promtail + Grafana, а не Prometheus, поскольку Loki специально разработан для работы с логами: он эффективно индексирует данные по лейблам, поддерживает полнотекстовый поиск через LogQL и интегрируется с Grafana для единого интерфейса с метриками из Prometheus. Prometheus же оптимален для числовых метрик, и попытка хранить в нём логи приведёт к неэффективному использованию ресурсов и ограниченным возможностям поиска.

## Задача 2: Логи

Выбор: Loki + Grafana + Promtail

Всё подходит под задание. Promtail шарит логи с хостов в Loki, умеет собирать из stdout (docker/systemd). Если сеть упала — не страшно, там batch + retry + диск-буфер. Поиск через LogQL (фильтры по лейблам вроде service="auth" и текст). Графана как UI — можно шарить ссылки на конкретные логи. Для микросервисов: Loki индексирует только лейблы, а не всё подряд — быстро и экономно. В графане метрики из Prometheus и логи рядом, удобно дебажить. Лейблами можно разграничивать команды и окружения. ELK не подходит, потому что он жрёт много памяти/диска и сложнее в настройке (три компонента). Но если нужен прям мощный full-text поиск по всем полям — ELK.

## Задача 3: Мониторинг

Выбор: Prometheus + Grafana + Node Exporter

Почему так: Node Exporter собирает стандартные метрики с хостов (CPU, память, диск, сеть). Для своих сервисов можно добавлять кастомные метрики через клиентские библиотеки (Counter, Gauge и т.д.). Промокверим через PromQL, дашборды в Графане — всё шарится ссылками. Для микросервисов удобно - сам находит сервисы в Kubernetes/Docker, лейблы позволяют агрегировать по service, env, есть Alertmanager для алертов и recording rules для тяжёлых запросов. Не подходит Datadog/New Relic,т.к. дорого при росте, вендорлок, данные уходят наружу,а также Zabbix ведь он опрашивает по расписанию, для динамических микросервисов неудобно, да и PromQL проще. Но если инфраструктура классическая и статичная, то Zabbix.
